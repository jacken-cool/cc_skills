# 飞书格式测试点生成工作流

**触发关键词**：`生成测试点`、`创建测试点`、`生成飞书测试点`、`生成飞书格式测试点`、`生成mm文件`、`导入飞书思维笔记`

**工作流简介**：基于需求文档或脑图节点，生成可导入飞书思维笔记的测试点设计文档（支持 Markdown 层级格式和 .mm 导入格式）。覆盖测试范围、核心业务链路、各端测试点、高优先级场景、待确认项等标准模块结构。

---

## 执行清单规划

在开始执行前，工作流需要：

1. 使用 `todo_write` 工具创建执行清单，将下述执行步骤转化为 TODO 列表
2. 执行应按列表顺序进行，禁止跳过步骤
3. 每完成一个步骤后，更新对应 TODO 的状态为 `completed`
4. 开始新步骤时，将对应 TODO 状态更新为 `in_progress`

**标准 TODO 清单模板**：

```json
[
  {"id":"1", "status":"pending", "content":"确认参考节点与需求来源"},
  {"id":"2", "status":"pending", "content":"查找关联需求"},
  {"id":"3", "status":"pending", "content":"需求分析"},
  {"id":"4", "status":"pending", "content":"检索需求知识库"},
  {"id":"5", "status":"pending", "content":"生成飞书Markdown测试点"},
  {"id":"6", "status":"pending", "content":"生成.mm格式文件（如需要）"},
  {"id":"7", "status":"pending", "content":"添加节点到脑图（如需要）"}
]
```

## 执行流程

### 步骤 1：确认参考节点与需求来源

根据实际用户意图选择合适方式：

- **方式一**：从对话上下文中读取（SKILL 加载时已执行 load_session，结果在上下文中）
- **方式二**：使用 `tools/load_session.md` 工具读取会话信息
- **方式三**：使用 `tools/search_nodes.md` 工具从脑图节点查询
- **方式四**：若用户提供了需求文档文件路径，直接读取该文档

节点信息包含：uid, name, kind, instance；若未找到则提示用户并终止。

### 步骤 2：查找关联需求

根据实际用户意图选择合适方式：

- **方式一**：从对话上下文中读取
- **方式二**：使用 `tools/load_session.md` 工具
- **方式三**：使用 `tools/search_nodes.md` 工具从脑图查询 STORY 或 BUG 类型节点
- **方式四**：使用步骤 1 读取到的需求文档作为关联需求

如未找到需求节点信息，提示用户并终止。

### 步骤 3：需求分析

- **使用工具**：`tools/select_rule.md`，传入关键词"需求分析"
- **根据返回结果**：
  - 返回自定义规则路径（`.codebuddy/rules/xxx`）：使用 `read_rules` 读取
  - 返回默认 Generator 路径（`generators/xxx`）：使用 `read_file` 读取
- **执行方式**：按获取到的规则 / Generator 定义执行需求分析
- **复用逻辑**：如该需求已分析过，直接使用历史分析文档

### 步骤 4：检索需求知识库

- **使用工具**：`tools/rag_search.md`
- **检索目标**：从步骤 3 的需求分析中提取 3-5 个核心关键词进行检索
- **结果合并**：将检索内容追加到需求分析文档末尾 "## 知识库参考信息" 章节
- **说明**：工具内部自动判断是否有可用知识库，无需手动判断

### 步骤 5：生成飞书 Markdown 格式测试点

基于需求分析结果，输出**飞书思维笔记可直接整理的层级结构**。

**测试点模块结构（必须覆盖）**：

```markdown
# [需求名称] 测试点

## 1. 测试范围
### 范围内
- 模块A
- 模块B
### 范围外
- 非本次改造项

## 2. 核心业务链路
### 配置端
- 字段新增
- 默认值
- 校验规则
### 同步链路
- 新增同步
- 编辑同步
- 删除同步
### 展示端
- 召回
- 过滤
- 排序
- 跳转

## 3. 前端 / APP 测试点
### tab 展示
- 默认态
- 切换态
### 搜索结果
- 名称召回
- 描述仅展示
- icon 兜底
### 异常态
- 无结果
- 网络异常
- 服务异常

## 4. 配置后台测试点
- 必填校验
- 长度校验
- 开关校验
- 回填校验

## 5. 运营后台测试点
- tab 管理
- 菜单管理
- 有效期
- 人群

## 6. 跨系统同步测试点
（根据需求内容填写）

## 7. 搜索中心 / 服务端规则测试点
（根据需求内容填写）

## 8. 异常与回归测试点
- 服务降级
- 网络异常
- 并发场景
- 回归影响面

## 9. 高优先级场景
### P0
- 主流程闭环
- 权限过滤
- 有效期过滤
### P1
- 边界值
- 输入校验
- 异常补偿
### P2
- 回归稳定性

## 10. 待确认项
- 计数口径
- 冲突优先级
- 刷新时机
```

**输出要求**：
- 输出完整的 Markdown 文件，命名建议：`xxx_test_points_feishu.md`
- 根据实际需求内容填充每个模块，不相关的模块可标注"本次不涉及"

### 步骤 6：生成 .mm 格式文件（若用户要求可导入飞书思维笔记的 .mm 文件）

基于步骤 5 的 Markdown 内容，转换为 `.mm` 格式文件。

**格式规范**（参考 `references/mm_mindnote_format.md`）：

```xml
<?xml version="1.0" encoding="utf-8"?>
<map>
  <node ID="root" TEXT="需求名称">
    <node ID="{md5(text)}" TEXT="测试范围" STYLE="bubble" POSITION="right">
      <node ID="{md5(text)}" TEXT="范围内" STYLE="fork">
        <node ID="{md5(text)}" TEXT="模块A" STYLE="fork"/>
      </node>
    </node>
  </node>
</map>
```

**关键规则**：
1. **ID 生成**：`hashlib.md5(text.encode()).hexdigest()`，根节点固定为 `"root"`
2. **一级节点**：`STYLE="bubble"`，需设置 `POSITION="left"` 或 `POSITION="right"`（平均分配左右）
3. **二级及以下**：`STYLE="fork"`，不设置 POSITION
4. 文件编码为 UTF-8

**一级节点左右分配建议**：

| POSITION | 模块 |
|---------|------|
| right | 测试范围、核心业务链路、前端/APP测试点、搜索/服务端测试点、待确认项 |
| left | 配置后台测试点、运营后台测试点、跨系统同步测试点、异常与回归测试点、高优先级场景 |

**输出要求**：
- 生成完整的 .mm 文件，命名建议：`xxx_test_points_feishu.mm`
- 可直接通过飞书思维笔记「导入」功能导入

**参考模板**：`assets/test_points_feishu_template.mm`

### 步骤 7：添加节点到脑图（如会话中存在脑图上下文）

- **使用工具**：`tools/add_nodes.md`
- **说明**：将生成的测试点节点添加到脑图并渲染
- 若无脑图上下文，跳过此步骤，只输出文件
