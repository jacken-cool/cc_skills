# 测试用例生成工作流

**触发关键词**：`生成测试用例`、`生成用例`、`生成CSV用例`、`根据测试点生成用例`、`根据需求生成用例`、`导出用例`、`生成markdown用例`

**工作流简介**：基于需求文档或测试点，生成测试用例文件。**默认输出 Markdown 格式**（按模块分组表格，可直接阅读评审），用户明确要求 CSV 时才输出 CSV。

---

## 执行清单规划

在开始执行前，工作流需要：

1. 使用 `todo_write` 工具创建执行清单，将下述执行步骤转化为 TODO 列表
2. 执行应按列表顺序进行，禁止跳过步骤
3. 每完成一个步骤后，更新对应 TODO 状态为 `completed`
4. 开始新步骤时，将对应 TODO 状态更新为 `in_progress`

**标准 TODO 清单模板**：

```json
[
  {"id":"1", "status":"pending", "content":"确认参考节点与需求来源"},
  {"id":"2", "status":"pending", "content":"查找关联需求"},
  {"id":"3", "status":"pending", "content":"需求分析"},
  {"id":"4", "status":"pending", "content":"检索需求知识库"},
  {"id":"5", "status":"pending", "content":"生成测试用例"},
  {"id":"6", "status":"pending", "content":"输出CSV文件"},
  {"id":"7", "status":"pending", "content":"添加节点到脑图（如需要）"}
]
```

## 执行流程

### 步骤 1：确认参考节点与需求来源

根据实际用户意图选择合适方式：

- **方式一**：从对话上下文中读取（SKILL 加载时已执行 load_session，结果在上下文中）
- **方式二**：使用 `tools/load_session.md` 工具读取会话信息
- **方式三**：使用 `tools/search_nodes.md` 工具从脑图节点查询
- **方式四**：若用户提供了需求文档或测试点文档文件路径，直接读取

节点信息包含：uid, name, kind, instance；参考节点可能是单个或多个。若未找到则提示用户并终止。

### 步骤 2：查找关联需求

根据实际用户意图选择合适方式：

- **方式一**：从对话上下文中读取
- **方式二**：使用 `tools/load_session.md` 工具
- **方式三**：使用 `tools/search_nodes.md` 工具从脑图查询 STORY 或 BUG 类型节点
- **方式四**：使用步骤 1 读取到的需求 / 测试点文档作为关联需求

如未找到需求节点信息，提示用户并终止。

### 步骤 3：需求分析

- **使用工具**：`tools/select_rule.md`，传入关键词"需求分析"
- **根据返回结果**：
  - 返回自定义规则路径（`.codebuddy/rules/xxx`）：使用 `read_rules` 读取
  - 返回默认 Generator 路径（`generators/xxx`）：使用 `read_file` 读取
- **执行方式**：按获取到的规则 / Generator 定义执行需求分析
- **复用逻辑**：如该需求已分析过，直接使用历史分析文档

### 步骤 4：检索需求知识库

- **使用工具**：`tools/rag_search.md`
- **检索目标**：从步骤 3 的需求分析中提取 3-5 个核心关键词进行检索
  - 核心功能模块（如"用户登录"、"权限管理"、"数据导出"）
  - 关键业务术语和专业词汇
  - 技术要点（如"API接口"、"数据库事务"、"缓存策略"）
- **检索策略**：优先 2-3 个关键词组合检索；不足时针对单个重要关键词补充检索
- **结果合并**：将检索内容追加到需求分析文档末尾 "## 知识库参考信息" 章节
- **说明**：工具内部自动判断是否有可用知识库

### 步骤 5：生成测试用例

基于步骤 3+4 的完整需求分析，生成测试用例列表。

**生成规则**：

1. **先覆盖 P0**：主流程闭环、权限校验、关键业务链路。
2. **再补 P1**：边界值、字段校验、异常场景、跨系统联动。
3. **最后补 P2**：回归稳定性、兼容性、性能场景。
4. **单条用例原则**：一件事一条用例，步骤清晰，预期明确。
5. **批量任务**：不同链接 / 导航之间的编号、模块、关联需求可区分。

**默认输出格式：Markdown 表格**（按模块分组）

```markdown
## 模块名称

| 用例标题 | 等级 | 前置条件 | 操作步骤 | 预期结果 |
|---------|:---:|---------|---------|---------|
| xxx | P0 | xxx | 1. xxx | 1. xxx |
```

- 每个模块一个二级标题
- 表格列：用例标题、等级、前置条件、操作步骤、预期结果
- 操作步骤和预期结果中使用 `1.` `2.` `3.` 分点

**如需 CSV 格式，用户明确要求后才输出：**

**标准字段结构**（固定字段顺序，仅保留7个核心字段，其他置空）：

| 字段 | 说明 |
|-----|------|
| 用例标题 | 简洁描述测试目标，如"正常登录-用户名密码均正确" |
| 用例目录 | 层级结构，如"用户登录/密码验证" |
| 等级 | P0 / P1 / P2 |
| 前置条件 | 执行该用例前需满足的状态，如"用户已注册且账号正常" |
| 步骤描述类型 | 固定填写"步骤" |
| 步骤描述 | 使用 `1.` `2.` `3.` 多行分点格式，每步单独一行 |
| 预期结果 | 使用 `1.` `2.` `3.` 多行分点格式，与步骤一一对应 |

**其他字段（置空）**：用例ID、UUID、是否自动化、评估工时、标签、关联需求、关联的自动化用例、执行次数、创建人、创建时间

**步骤描述与预期结果格式示例**：

```
步骤描述：
1. 打开登录页
2. 输入正确的用户名和密码
3. 点击"登录"按钮

预期结果：
1. 登录页正常展示，用户名和密码输入框可见
2. 输入内容正确回填，无错误提示
3. 登录成功，跳转到首页，顶部显示用户头像和昵称
```

### 步骤 6：输出测试用例文件

**默认输出 Markdown 格式**（当用户未明确要求 CSV 时）。

#### 6a：输出 Markdown（默认）

将步骤 5 生成的用例整理为 Markdown 格式，按模块分组：

```markdown
# XXX测试用例 — 功能名称

共 N 条用例

## 模块A

| 用例标题 | 等级 | 前置条件 | 操作步骤 | 预期结果 |
|---------|:---:|---------|---------|---------|
| xxx | P0 | 1. xxx | 1. xxx | 1. xxx |

## 模块B
...
```

**Markdown 输出要求**：
1. 文件命名：`xxx_test_cases.md`
2. 按模块分组，每个模块一个二级标题 + 一个表格
3. 表格列固定：用例标题、等级、前置条件、操作步骤、预期结果
4. 操作步骤和预期结果用 `1.` `2.` `3.` 分点
5. 文件编码：UTF-8（直接使用 Write 工具或 Python 均可）

#### 6b：输出 CSV（仅用户明确要求时）

当用户明确要求 CSV 格式时，将步骤 5 生成的用例整理为标准 CSV 格式输出。

**⚠️ 重要：CSV 必须使用 UTF-8 BOM 编码（避免 Excel 乱码）**

**⚠️ 重点：必须使用 Python csv.writer 写入，禁止用 Write 工具直接写 CSV**

Python 的 `csv.writer` 会自动处理：
- 双引号包裹含逗号的字段
- 字段内换行符的正确转义

如果用 Write 工具，容易产生两个隐蔽问题：
1. `\n` 被写入为字面文本而非换行（操作步骤和预期结果不会真正换行）
2. 字段中含英文逗号 `,` 时导致列偏移（如 `(500,T2)` 被解析为 3 列）

**推荐执行方式（Python 脚本）**：

由于 `write_to_file` 工具默认不带 BOM，请使用 `execute_command` 调用 Python 脚本写入：

```python
# 写入CSV文件（UTF-8 BOM编码，Excel可直接打开）
import csv

output_path = 'xxx_test_cases.csv'
with open(output_path, 'w', encoding='utf-8-sig', newline='') as f:
    writer = csv.writer(f)
    # 写入表头
    writer.writerow(['用例ID','UUID','用例标题','用例目录','是否自动化','等级','前置条件','步骤描述类型','步骤描述','预期结果','评估工时','标签','关联需求','关联的自动化用例','执行次数','创建人','创建时间'])
    # 写入数据行（示例）
    writer.writerow(['', '', '正常登录-用户名密码均正确', '用户登录/密码验证', '', 'P0', '用户已注册且账号正常', '步骤', '1. 打开登录页\n2. 输入正确用户名和密码\n3. 点击登录按钮', '1. 登录页正常展示\n2. 输入内容正确回填\n3. 登录成功跳转首页', '', '', '', '', '', '', ''])
```

**execute_command 示例**：
```
python.exe -c "import csv; f=open('xxx_test_cases.csv','w',encoding='utf-8-sig',newline=''); w=csv.writer(f); w.writerow([...]); ..."
```

**⚠️ 禁止直接用 write_to_file 工具写入 CSV**（除非确认该工具已支持 UTF-8 BOM）。

**CSV 格式要求**：

1. 第一行为表头：`用例ID,UUID,用例标题,用例目录,是否自动化,等级,前置条件,步骤描述类型,步骤描述,预期结果,评估工时,标签,关联需求,关联的自动化用例,执行次数,创建人,创建时间`
2. **仅填写7个核心字段**（用例标题、用例目录、等级、前置条件、步骤描述类型、步骤描述、预期结果），其他字段置空
3. 步骤描述和预期结果中使用**真正的换行符**（不是 `\n` 文本）
4. 字段值中若包含逗号，使用双引号包裹整个字段
5. **文件编码必须为 UTF-8 with BOM（`utf-8-sig`）**，这是 Excel 正确显示中文的关键
6. 文件命名建议：`xxx_test_cases.csv`

**CSV 输出示例**：

```csv
用例ID,UUID,用例标题,用例目录,是否自动化,等级,前置条件,步骤描述类型,步骤描述,预期结果,评估工时,标签,关联需求,关联的自动化用例,执行次数,创建人,创建时间
,,正常登录-用户名密码均正确,用户登录/密码验证,,P0,用户已注册且账号正常,步骤,"1. 打开登录页
2. 输入正确用户名和密码
3. 点击登录按钮","1. 登录页正常展示
2. 输入内容正确回填
3. 登录成功跳转首页",,,,,,,,
,,异常登录-密码错误,用户登录/密码验证,,P1,用户已注册且账号正常,步骤,"1. 打开登录页
2. 输入正确用户名+错误密码
3. 点击登录按钮","1. 登录页正常展示
2. 输入内容正确回填
3. 提示「密码错误」，不跳转",,,,,,,,
```

**批量任务命名**：`01_xxx_test_cases.csv`、`02_xxx_test_cases.csv`

## 质量检查项

**Markdown 输出检查：**
- [ ] 所有 P0 用例覆盖了主流程的每个关键节点
- [ ] 按模块分组，每个模块一个表格
- [ ] 操作步骤与预期结果均使用分点格式
- [ ] 批量任务中的模块命名口径一致
- [ ] 待确认项对应的用例已标注"待确认"或暂不生成

**CSV 输出检查（仅生成CSV时）：**
- [ ] 所有 P0 用例覆盖了主流程的每个关键节点
- [ ] 步骤描述与预期结果均使用分点格式
- [ ] 仅填写7个核心字段，其他字段已置空
- [ ] 批量任务中的模块命名口径一致
- [ ] 待确认项对应的用例已标注"待确认"或暂不生成
